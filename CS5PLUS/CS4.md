## Общая оценка

Проект представляет собой **калькулятор на Vanilla JS** с поддержкой базовых и расширенных операций, историей вычислений, пакетным режимом через **Web Worker** и модульной логикой для Node/Jest.
По результатам анализа видно, что работа выполнена качественно: архитектура в целом понятная, тесты на базовую логику есть, приложение можно быстро запустить без сложной сборки.

При этом выявлены и слабые места: часть заявленных возможностей реализована не полностью последовательно, есть расхождения между браузерной и тестируемой логикой, а покрытие тестами ограничено в основном вычислительным ядром.

Отдельно отмечу, что **автотесты проходят успешно**: 7 из 7 тестов завершились без ошибок.

# 1. Функциональность

## Что реализовано хорошо

Программа поддерживает:

* арифметические операции `+`, `-`, `*`, `/`
* дополнительные операции `^` и `%`
* работу с отрицательными и десятичными числами
* историю вычислений через `localStorage`
* пакетные вычисления с массивами
* фоновую обработку через `Web Worker`
* очистку результата, полей и истории
* обработку ошибок, включая деление на ноль и некорректные значения

С точки зрения основной бизнес-логики калькулятор выполняет свои задачи корректно.

## Выявленные проблемы

### 1. Неоднозначная обработка чисел с запятой

В коде заявлена поддержка десятичных чисел и нормализация запятой в точку, но фактически строка вида:

* `1,2` интерпретируется как массив `[1, 2]`, а не как число `1.2`
* `-1,5` интерпретируется как массив `[-1, 5]`, а не как `-1.5`

Это функциональная ошибка, потому что пользователь в русскоязычном интерфейсе естественно ожидает поддержку десятичной запятой.

### 2. Дублирование бизнес-логики

Логика калькулятора реализована:

* отдельно в `src/Calculator.js`
* отдельно в `public/app.js`
* отдельно в `public/worker.js`

Из-за этого одна и та же функциональность поддерживается в нескольких местах. Это повышает риск расхождения поведения между тестируемой логикой и логикой, которую реально использует браузер.

### 3. Ошибки локализации

Часть сообщений в браузере на русском, а часть логики и ошибок в `worker.js` и `src` — на английском. Пользовательский опыт из-за этого получается не полностью согласованным.

## Оценка

**Функциональность: хорошая, но с заметными недочетами.**

# 2. Производительность

## Плюсы

* Для пакетных вычислений используется **Web Worker**, что правильно: тяжелые операции не блокируют интерфейс.
* Есть fallback-режим без worker, что повышает устойчивость приложения.
* DOM-запросы к ключевым элементам кэшируются.
* Обновление UI организовано достаточно бережно.

## Ограничения

* Для текущего масштаба проекта производительность хорошая, но реальных замеров нет.
* Нет benchmark-тестов для больших массивов.
* История при каждом обновлении перерисовывается полностью; при лимите 100 записей это некритично, но архитектурно не идеально.

## Оценка

**Производительность: хорошая для учебного/небольшого прикладного проекта.**

# 3. Удобство использования (юзабилити)

## Что сделано удачно

* Интерфейс понятный и визуально аккуратный.
* Есть явные поля для ввода, выбора операции и настройки точности.
* Ошибки подсвечиваются.
* Есть история вычислений.
* Поддерживается Enter для запуска расчета.
* Интерфейс адаптирован под мобильные устройства.

## Недочеты

### 1. Неочевидность формата ввода

Пользователь может не понять, когда запятая — это разделитель массива, а когда десятичный разделитель.

### 2. Недостаточно явная валидация

Символьная фильтрация не предотвращает все некорректные конструкции на этапе ввода, а лишь часть из них.

### 3. Ограниченная доступность

Есть базовые ARIA-атрибуты, но для полноценной accessibility-поддержки не хватает:

* более явного связывания ошибок с полями
* улучшенной клавиатурной навигации
* более полной семантики сообщений об ошибках

## Оценка

**Юзабилити: выше среднего, но требует доработки форматов ввода и доступности.**

# 4. Безопасность

## Плюсы

* Не используется `innerHTML` для пользовательских строк.
* История сохраняется в `localStorage`, а не отправляется куда-либо.
* Для данного класса приложения поверхность атаки небольшая.

## Слабые места

* Валидация реализована только на клиенте.
* Нет политики Content Security Policy.
* Нет механизма безопасной интеграции, если приложение будет подключаться к серверу в будущем.
* Нет защиты от злоупотребления чрезмерно большими входными массивами.

## Оценка

**Безопасность: приемлемая для локального клиентского приложения, но недостаточная для промышленного развертывания.**

# 5. Масштабируемость

## Сильные стороны

* Есть разделение на модули `Calculator`, `EventBus`, `utils`.
* Операции вынесены в map-структуру, что удобно для расширения.
* Подход с worker подходит для дальнейшего роста нагрузки.

## Проблемы

* Архитектура масштабируется хуже из-за дублирования логики между `src`, `public/app.js` и `worker.js`.
* Нет единого слоя доменной логики, который переиспользуется в UI и в worker.
* Отсутствует сборка/модульная упаковка, поэтому расширение проекта будет усложняться.

## Оценка

**Масштабируемость: средняя. Основа есть, но текущая структура начнет мешать при росте проекта.**

# 7. Переносимость

## Плюсы

* Приложение запускается как обычный статический сайт.
* Нет тяжелых зависимостей.
* Node/Jest используются только для тестирования.
* В целом решение переносимо между системами.

## Ограничения

* Кроссбраузерность заявлена, но полноценно не подтверждена тестированием.

* Web Worker может вести себя по-разному в специфических окружениях без дополнительной проверки.

## Оценка

**Переносимость: хорошая для современных браузеров.**

# Итоговый вывод

Работа выполнена на хорошем уровне: приложение рабочее, логика в целом корректна, тесты на ядро есть, интерфейс понятен, архитектура уже содержит полезные инженерные решения вроде Web Worker и модульного разбиения.

Основные проблемы не в том, что программа “не работает”, а в том, что она **не полностью доведена до консистентного и расширяемого состояния**. Главные риски:

* неоднозначный ввод десятичных чисел с запятой
* дублирование бизнес-логики
* ограниченное тестирование UI и браузерной части
* неполная готовность к масштабированию

# Предложения по решению выявленных проблем

## 1. Исправить разбор десятичных чисел и массивов

Нужно четко разделить:

* десятичную запятую внутри одного числа
* разделители элементов массива

### Рекомендуемый вариант

Поддерживать:

* `1.5` и `1,5` как скаляр
* массивы вводить только через пробел или `;`
* либо добавить отдельный переключатель “режим массива”

Это устранит неоднозначность.

---

## 2. Убрать дублирование логики

Следует сделать единое вычислительное ядро и использовать его:

* в Node-тестах
* в браузере
* в worker

### Практическое решение

* оставить доменную логику только в `src`
* подключить сборщик или организовать общий модуль для браузера
* `worker.js` должен импортировать ту же вычислительную функцию, а не содержать собственную копию

Это резко улучшит функциональную согласованность и сопровождение.

---

## 3. Разделить уровни ответственности

Сейчас `app.js` перегружен.

### Лучше выделить:

* `parser.js`
* `calculator.js`
* `history.js`
* `ui.js`
* `worker-client.js`

Такой рефакторинг облегчит тестирование и расширение.

---

## 4. Расширить тестирование

Нужно добавить:

* unit-тесты для `utils`
* тесты парсинга ввода
* тесты истории
* тесты worker
* UI-тесты
* e2e-тесты

### Минимальный приоритет

Сначала покрыть:

1. `parseScalarOrArray`

## 5. Улучшить UX

* сделать подсказку различия между числом и массивом
* унифицировать все сообщения на русском
* добавить более точные сообщения об ошибках

---

## 6. Усилить безопасность

* добавить ограничения на размер входных массивов
