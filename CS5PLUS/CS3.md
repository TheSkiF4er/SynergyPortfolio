## 1) Краткий вывод

Проект выглядит как **хорошо оформленное учебное/MVP Django-приложение**: есть форма, CRUD, базовый JSON API, DRF API, валидация, кэширование, Docker-окружение, README и базовые тесты.
Для **демо/курсовой/пилотной версии** решение выполнено добротно.

При этом для **реальной production-эксплуатации** есть несколько критичных зон:

* слабая защита API;
* не до конца надежная обработка дубликатов;
* ограниченное тестовое покрытие;
* локальный кэш и SQLite как временные решения;
* несколько архитектурных и UX-несогласованностей.

## 3) Детальный обзор

### Функциональность

**Что сделано хорошо:**

* реализован основной сценарий: ввод имени, сохранение, вывод последних записей;
* есть редактирование и удаление записей;
* реализованы 2 API-варианта: простой JSON endpoint и DRF CRUD;
* есть валидация имени;
* применен PRG-паттерн и сообщения пользователю.

**Проблемы:**

1. В `views.py` параметр `limit` для API берется через `int(request.GET.get("limit", "20"))` без защиты от нечисловых и некорректных значений.
   Риск: при `limit=abc` или отрицательных значениях возможна ошибка/некорректное поведение.
2. Защита от дубликатов реализована только на уровне формы, а не БД.
   Риск: при параллельных запросах возможны гонки и дубликаты.
3. Несогласованность ограничений: в модели `name` до 100 символов, а в формах — до 50.

**Рекомендации:**

* валидировать `limit` через безопасный парсинг с fallback;
* вынести контроль уникальности в БД;
* унифицировать длину поля в модели/форме.

---

### Производительность

**Плюсы:**

* приложение небольшое, запросы простые;
* список ограничен, есть пагинация в ListView;
* используется кэширование последних записей и JSON API.

**Проблемы:**

1. Кэш `LocMemCache` подходит только для локальной/однопроцессной схемы.
   При нескольких worker-процессах кэш будет несогласован.
2. Инвалидация кэша частичная: сбрасывается `api_greetings:20`, но не другие возможные лимиты (`10`, `50` и т.д.).
3. DRF API не имеет пагинации, фильтрации, лимитов и throttling.
4. SQLite — нормальный выбор для учебного проекта, но слабый для роста нагрузки.

**Рекомендации:**

* перейти на Redis для кэша;
* изменить стратегию инвалидации (например, versioned cache key);
* включить пагинацию и лимиты в DRF;
* перейти на PostgreSQL.

---

### Удобство использования (юзабилити)

**Плюсы:**

* хороший базовый UI на Bootstrap;
* есть понятные сообщения об ошибках;
* есть блокировка кнопки и спиннер при отправке;
* есть список последних регистраций и страница управления.

**Замечания:**

1. Есть терминологическая неоднородность: `Users`, `Manage names`, `Greeting App`, при том что логика — скорее “greetings/registrations”.
2. Интерфейс и README частично в разных языках.
3. Нет поиска/фильтрации по списку записей.
4. Нет подтверждения/защиты от случайного удаления кроме отдельной страницы.

**Рекомендации:**

* унифицировать терминологию;
* выбрать один язык интерфейса или включить i18n;
* добавить поиск, сортировку, фильтрацию;
* улучшить UX удаления и пустых состояний.

---

### Безопасность

**Плюсы:**

* используется Django ORM;
* CSRF для форм включен;
* HTML в шаблонах по умолчанию экранируется;
* есть security-настройки для `DEBUG=False`;
* секреты вынесены в env-переменные.

**Критичные проблемы:**

1. **DRF API открыт с `AllowAny` на полный CRUD.**
   Любой клиент сможет создавать, менять и удалять записи.
2. Нет rate limiting / throttling.
   Возможен спам и abuse.
3. Есть дефолтный fallback `SECRET_KEY="django-insecure-change-me"`.
   Для учебного проекта допустимо, для production — нет.
4. Bootstrap подключается через CDN.
   Это зависимость от внешней сети и дополнительный supply-chain риск.

**Рекомендации:**

* включить аутентификацию и permissions для DRF;
* добавить throttling;
* убрать небезопасный fallback `SECRET_KEY` в production;
* при необходимости хранить фронтенд-ресурсы локально;
* добавить журналирование действий и аудит API.

---

### Масштабируемость

**Плюсы:**

* структура проекта позволяет расширять функциональность;
* есть разделение на views/forms/models/serializers/api;
* предусмотрены env-настройки БД и кэша.

**Проблемы:**

1. SQLite и LocMemCache ограничивают рост.
2. Нет DB-ограничений для целостности данных.
3. Нет сервисного слоя — логика пока сконцентрирована в view/form.
4. Два API-подхода одновременно (ручной JSON + DRF) увеличивают сложность развития.

**Рекомендации:**

* перейти на PostgreSQL + Redis;
* добавить ограничения и индексы;
* вынести бизнес-логику в service layer;
* оставить один основной API-стек либо четко развести роли двух API.

---

### Сопровождаемость

**Плюсы:**

* код разбит на модули;
* есть docstring-и;
* README объясняет запуск и roadmap;
* есть Docker и пример `.env`.

**Проблемы:**

1. Дублируется логика валидации имени в `GreetingForm` и `GreetingModelForm`.
2. Нет CI-конфигурации.
3. Нет логирования и наблюдаемости.
4. Нет проверки типов и автоматизированного quality gate.

**Рекомендации:**

* вынести валидацию в отдельный validator/helper;
* добавить CI (tests + lint);
* внедрить logging;
* добавить `mypy`/`ruff`/`black` в pipeline.

---

### Переносимость

**Плюсы:**

* есть Dockerfile;
* настройки через env;
* приложение можно запускать локально и в контейнере.

**Замечания:**

1. `docker-compose.yml` ожидает `.env`, но в проекте только `.env.example`.
2. Зависимость от CDN ухудшает автономность.
3. Не видно матрицы проверки на разных Python/OS.

**Рекомендации:**

* добавить шаг копирования `.env.example -> .env` в инструкцию;
* при необходимости убрать зависимость от CDN;
* протестировать на Python 3.11/3.12.

## 4) Основные выявленные проблемы и способы их решения

### Приоритет 1 — безопасность API

**Проблема:** DRF CRUD открыт без авторизации.
**Решение:**

* добавить authentication;
* ограничить `POST/PUT/PATCH/DELETE`;
* включить permissions;
* настроить throttling.

---

### Приоритет 2 — надежность данных

**Проблема:** защита от дубликатов только в форме.
**Решение:**

* добавить DB-constraint на уникальность;
* нормализовать имя перед сохранением;
* обрабатывать `IntegrityError`.

---

### Приоритет 3 — устойчивость API

**Проблема:** `limit` не валидируется безопасно.
**Решение:**

* сделать безопасный парсинг:

  * если не число → 20;
  * если < 1 → 1;
  * если > 100 → 100.
* добавить тесты на эти кейсы.

---

### Приоритет 4 — масштабирование и производительность

**Проблема:** SQLite + LocMemCache + неполная инвалидация кэша.
**Решение:**

* PostgreSQL;
* Redis;
* versioned cache keys или паттерн-очистка;
* пагинация в DRF.

---

### Приоритет 5 — сопровождение

**Проблема:** дублирование валидации и отсутствие CI.
**Решение:**

* вынести валидацию в общий validator;
* настроить GitHub Actions / CI:

  * lint,
  * tests,
  * format check.

## 5) Итоговое заключение

**Итог:** работа выполнена на хорошем уровне для учебного проекта или MVP.
Проект уже демонстрирует:

* рабочую функциональность;
* понятную структуру;
* приемлемый UX;
* базовую инженерную дисциплину.

Но для уровня **production-ready** необходимо в первую очередь доработать:

1. безопасность API;
2. целостность данных на уровне БД;
3. тестирование и CI;
4. кэш/БД под рост нагрузки;
5. архитектурную унификацию и устранение дублирования.
